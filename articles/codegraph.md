## はじめに

Claude CodeやCursorを使った開発で、誰もが直面する課題がある。中規模以上のプロジェクトで「この処理の流れを追ってほしい」と依頼すると、AIはプロジェクト全体を探索し始める。数十ファイルの読み込み、数百回の検索、数百万Tokenの消費。APIの請求書を見るたびに胃が痛くなる。

さらに深刻なのは、これだけのリソースを消費しても、AIが「見つかりませんでした」と答えるケースだ。広大なコードベースの中で方向感覚を失い、Tokenだけが無駄に消えていく。

[CodeGraph](https://github.com/colbymchenry/codegraph)はこの問題を根本から解決する。32,100 GitHub Star、MITライセンス。AIを賢くするのではなく、AIにコードベースの「地図」を渡すアプローチだ。

## 対象読者

- Claude Code、Cursor、Codexを日常的に使う開発者
- Token消費とAPIコストに悩むエンジニア
- 大規模コードベースでのAI活用効率を改善したい方

## この記事を読むメリット

- CodeGraphの仕組みとセマンティック検索との根本的な違いを理解できる
- 7プロジェクトのベンチマークから具体的な削減効果を把握できる
- tree-sitter + SQLite + MCPの技術スタックの設計意図がわかる
- 導入から初期設定までの手順と注意点が明確になる

## 結論

CodeGraphはAIコーディングのボトルネックが「モデルの知能」ではなく「コードベース理解の効率」にあることを示したプロジェクトだ。tree-sitterによる構造解析とSQLiteベースの知識グラフにより、平均57%のToken削減、35%のコスト削減、71%の操作削減を達成している。完全ローカル動作でデータが外部に送信されない点も、実運用上の大きなアドバンテージである。このツールの本質は「モデルを賢くする」ことではなく「無駄な探索を省く」ことにある。

## 仕組み：Cursorとの根本的な違い

CodeGraphがCursorのインデックスと根本的に異なるのは、提供する情報の「質」だ。

Cursorはベクトル埋め込みによるセマンティック検索を使う。質問をベクトル化し、コードベース内で類似度の高い断片をランキング形式で返す。模糊探索には有効だが、決定的な弱点がある——**コードの構造を理解しない**。

関数Aが関数Bを呼び出し、モジュールCからインポートしている——このような関係性は、セマンティック検索では得られない。AIは「関連しそうな断片」のリストを受け取り、それを手がかりに自分でファイルを開き、関係性を検証しなければならない。

CodeGraphは異なる。**tree-sitter**でソースコードを解析し、明示的なコールグラフを構築する。tree-sitterはGitHubがAtomエディタ用に開発した増分構文解析ライブラリで、現在はNeovimのコア構文エンジンとして採用されている。従来のコンパイラフロントエンドと異なり、耐障害性（フォールトトレランス）に優れ——コードに構文エラーがあっても部分的なASTを生成できる——解析速度も速い。

抽出されたシンボル（関数、クラス、メソッド）とエッジ（呼び出し、インポート、継承）は、すべてローカルのSQLiteデータベース（FTS5全文検索付き）に保存される。

AI Agentからの問い合わせに対しては、プロジェクト全体のファイル探索ではなく、知識グラフへのSQLクエリで応答する。1回のMCPツール呼び出しでエントリポイント・関連シンボル・コードスニペットが返却される。

## 他のコードインテリジェンスツールとの比較

**Cursorのセマンティック検索**：探索には便利だが、呼び出し関係の確定情報は得られない。AIが自分で検証する必要がある。

**Gemini Code Assist**：Googleのクラウドインフラで超大規模リポジトリを解析可能だが、コードがサーバーにアップロードされる。規制業界では致命的な制約。

**Sourcegraph**：高機能だが重い。サーバーデプロイ、インデクサー設定、インフラ維持が必要。CodeGraphは単一バイナリとSQLiteファイルで完結する。

**GitHub Copilotコードベースインデックス**：ベータ段階、クラウドのみ、限定的ロールアウト。

CodeGraphの差別化は「**ローカル＋軽量＋構造グラフ**」の3点にある。

## ベンチマーク結果

7言語・7つの実在OSSプロジェクトで、Claude Opus 4.7（ヘッドレスモード `claude -p`）を使った比較テストの結果。同一タスク・同一モデルで、CodeGraphの有無のみを変数とした4回の実行から中央値を採用：

| プロジェクト | 言語 | ファイル数 | Token削減 | コスト削減 | 高速化 | 操作削減 |
|-------------|------|-----------|----------|----------|--------|---------|
| VS Code | TypeScript | ~10,000 | 78% | 26% | 52% | 85% |
| Excalidraw | TypeScript | ~640 | 90% | 52% | 73% | 96% |
| Tokio | Rust | ~790 | 86% | 82% | 71% | 92% |
| Django | Python | ~3,000 | 36% | 12% | 19% | 53% |
| Alamofire | Swift | ~110 | 64% | 47% | 48% | 83% |
| OkHttp | Java | ~645 | 13% | 2% | 31% | 45% |
| Gin | Go | ~110 | 34% | 21% | 27% | 40% |

分析すべきパターンが3つある：

**第1に、プロジェクト規模と効果は比例する。** VS Code（約10,000ファイル）ではTokenが280万→60.1万。Excalidrawでは350万→34.4万。ファイル数が指数関数的に増えると、地図なしの探索コストも指数関数的に跳ね上がる。

**第2に、Rustプロジェクトで特に効果が大きい。** Tokioでは$2.41→$0.42と82%のコスト削減。Rustのモジュールシステム（`mod`、`use`、`pub use`、re-exportの多重階層）はAIの探索を特に消耗させる——CodeGraphはその迷路を一瞬で解決する。

**第3に、小規模プロジェクトでは効果が限定的。** OkHttp（Java, 645ファイル）ではわずか13%のToken削減。Gin（Go, 110ファイル）でも34%。プロジェクトが小さいうちは、AIが直接ファイルを読んでもグラフを参照しても大きな差が出ない。

重要なのは、これらがモデルの性能向上ではないこと。Claude Opus 4.7は両方のテストで同一のモデル。削減されたTokenはすべて、機械的なファイル探索に浪費されていたものだ。

## フレームワークルーティング認識

14フレームワークのルーティングパターンをネイティブ認識。アノテーションベース（Spring、NestJS）、デコレータベース（Flask、FastAPI）、DSLベース（Rails、Laravel）、ファイル規約ベース（Django URLconf、SvelteKit）——各パラダイムに専用の認識ルールを実装。

「`POST /api/users`を処理するハンドラは？」という質問に対し、CodeGraphはURLパターンから直接ハンドラ関数を特定する。ミドルウェアチェーン、ルートインクルード、デコレータスタックを一気に解決する。バックエンド開発者にとって、この機能だけでも導入価値がある。

## 3層の自動同期

**第1層：OSネイティブファイルウォッチャー。** macOS FSEvents、Windows ReadDirectoryChangesW、Linux inotify。各OS最下層のファイル変更通知。ポーリング不要、CPU消費なし。

**第2層：デバウンスバッチング。** 2秒の静寂ウィンドウ内の変更を1回の増分同期に集約。リファクタリング時の連続保存でも、ファイルごとの再構築は発生しない。

**第3層：Stalenessフラグ＋再接続時高速同期。** 未同期ファイルを明示的にマーク。再接続時はファイルサイズ＋修正時刻＋コンテンツハッシュの高速比較で変更分のみ同期。

保存するだけ。手動リビルド不要。

## 完全ローカル動作

知識グラフは `.codegraph/codegraph.db`（単一SQLiteファイル）。クラウドコンポーネントなし、APIキー不要、データ送信ゼロ。

## インストール

```bash
# macOS / Linux
curl -fsSL https://raw.githubusercontent.com/colbymchenry/codegraph/main/install.sh | sh

# Windows PowerShell
irm https://raw.githubusercontent.com/colbymchenry/codegraph/main/install.ps1 | iex
```

npmも可：`npx @colbymchenry/codegraph`

プロジェクト初期化：

```bash
codegraph init -i
```

対応AIツール：Claude Code、Cursor、Codex CLI、OpenCode、Hermes Agent、Gemini CLI、Antigravity、Kiro（インストーラーが自動検出・設定）。

注意点：
- macOS：先に `xcode-select --install` を推奨（未インストール時は5〜10倍低速）
- デフォルト除外：`node_modules`、`vendor`、`dist`、`build`、`target`、`.venv`、`.gitignore`該当ファイル、1MB超ファイル
- 初回全量インデックスは超大規模プロジェクトで10分以上かかる場合があるが、一度限り

## 現時点の制限

- 初期インデックス時間はコードベース規模とディスクI/Oに依存
- 言語サポートに濃淡あり（TypeScript/Python/Rust/Goが最も充実、Objective-Cは部分サポート）
- 探索の補助であり推論の代替ではない——「なぜ遅いのか」「設計提案」には直接寄与しない

## まとめ

AIコーディングツールの進化はモデル性能の向上に集中してきた。しかし日常的な開発ワークフローの中で本当のボトルネックになっているのは、「コードベースをどれだけ速く理解できるか」というエンジニアリング効率の方だ。

50ファイルを読み込んで300万Tokenを消費する賢いモデルより、10ファイルで60万Tokenで済む「地図を持った」モデルの方が、開発現場では圧倒的に使いやすい。

CodeGraphはその地図を提供する。
